## 1. Gap Analysis Process

### Step 1: Audit Current Coverage

Before searching for papers, audit what the showcase already covers:

```
# Count tasks per category
find . -name metadata.json -not -path './.git/*' | sed 's|/[^/]*/[^/]*$||' | sort | uniq -c | sort -rn

# List all annotation types in use
grep -rh '"annotationTypes"' */*/metadata.json */*/metadata.json 2>/dev/null | sort | uniq -c | sort -rn

# List all domains
grep -rh '"domain"' */*/metadata.json 2>/dev/null | python3 -c "import sys,json; [print(d) for l in sys.stdin for d in json.loads(l.split(':',1)[1].strip().rstrip(','))]" | sort | uniq -c | sort -rn
```

### Step 2: Identify Coverage Gaps

Compare the audit results against:

1. **Potato's supported annotation types** (check `https://github.com/davidjurgens/potato` for latest):
   - Core: radio, multiselect, text, likert, span, pairwise
   - Advanced: span_linking (typed relations), video_annotation, bbox, polygon
   - New (v2.0+): segmentation masks, bounding box on PDFs, video object tracking, coreference chains, entity linking, conversation tree, triage, n-ary event annotation, LLM pre-annotation, agentic annotation

2. **Under-represented modalities**: Count tasks per modality — audio, video, multimodal, code, documents typically lag behind text.

3. **Under-represented domains**: Clinical NLP, legal NLP, code annotation, cross-lingual tasks, dialogue systems, agentic AI are common gaps.

4. **Under-represented task types**: Relation extraction, coreference, discourse parsing, temporal reasoning, error span annotation, safety evaluation.

## 2. Paper Search Strategy

### Primary Search Venues

Search proceedings from these top-tier venues (most recent 3 years):

**NLP/CL:**
- ACL, EMNLP, NAACL, EACL, COLING (main + findings)
- SemEval, *SEM, CoNLL (shared tasks — great for annotation)
- Workshop proceedings: DISRPT, CRAC, WMT, BioNLP, ClinicalNLP, LAW

**Computer Vision:**
- CVPR, ICCV, ECCV, WACV

**Machine Learning:**
- NeurIPS (Datasets & Benchmarks track), ICML, ICLR

**Information Retrieval:**
- SIGIR, CIKM, WSDM

**Speech/Audio:**
- INTERSPEECH, ICASSP, SLT

**Interdisciplinary:**
- AAAI, IJCAI, WWW, KDD, CHI

### Search on Specific Platforms

1. **ACL Anthology** (`https://aclanthology.org/`): Best for NLP. Search by venue+year, filter by "dataset" or "annotation" in title/abstract.

2. **Semantic Scholar** (`https://semanticscholar.org`): Use field-of-study filters. Sort by citation count for established benchmarks.

3. **Papers With Code** (`https://paperswithcode.com`): Browse by task type — directly shows which papers introduced datasets. Check "Datasets" tab for each task.

4. **HuggingFace Datasets** (`https://huggingface.co/datasets`): Browse trending datasets. Many link to papers.

5. **Google Scholar**: Use date ranges (`2022..2025`), site restrictions (`site:aclanthology.org`, `site:openreview.net`).

6. **OpenReview** (`https://openreview.net`): Search ICLR, NeurIPS submissions directly.

7. **arXiv** (`https://arxiv.org`): Search cs.CL, cs.CV, cs.AI categories. Use date filters.

### Shared Task Proceedings (High-Value Sources)

Shared tasks are ideal because they define annotation schemas explicitly:

- **SemEval** (annual): NER, relation extraction, sentiment, etc.
- **WMT** (annual): Translation quality, error spans
- **BioNLP/ClinicalNLP** workshops: Medical entity/relation extraction
- **CRAC** (coreference): Multilingual coreference shared task
- **DISRPT** (discourse): Discourse segmentation and relations
- **n2c2** (clinical NLP): De-identified clinical text challenges
- **TREC** (IR): Various retrieval and QA tasks

## 3. Paper Selection Criteria

### Must-Have Criteria (all required)

1. **Published dataset with annotation schema**: The paper must describe a concrete annotation task with labeled examples. Pure method papers without datasets are excluded.

2. **Clear annotation types**: The task must map to one or more Potato annotation types (radio, multiselect, text, likert, span, span_linking, pairwise, bbox, polygon, video_annotation).

3. **Peer-reviewed venue**: Published at a recognized venue (see venue list above). arXiv-only papers are acceptable if highly cited (100+ citations) or from established groups.

4. **Reproducible**: The dataset is publicly available (or at least the annotation schema is documented enough to create representative samples).

5. **Fills a gap**: The task addresses an under-represented annotation type, modality, domain, or Potato feature.

### Nice-to-Have Criteria (for ranking)

- High citation count (indicates community adoption)
- Dataset available on HuggingFace or GitHub
- Exercises a new Potato feature (span_linking, video tracking, etc.)
- Covers an under-represented language or domain
- From a shared task (well-defined annotation guidelines)
- Recent publication (2023-2025 preferred)

### Exclusion Criteria

- No clear annotation schema (e.g., self-supervised datasets)
- Annotation task is trivially simple (e.g., binary classification of short strings with no context)
- Duplicate coverage (already have 3+ tasks of the same type)
- Dataset is behind a paywall or requires IRB/DUA that blocks sample creation
- Task requires proprietary tools or non-Potato annotation interfaces

## 4. Paper-to-Task Mapping Process

### Step 1: Extract Annotation Schema

For each selected paper:
1. Read the annotation guidelines section
2. Identify the annotation types (radio, span, etc.)
3. List all label categories with definitions
4. Note any multi-step annotation workflows
5. Identify the data format (what fields does each item have?)

### Step 2: Map to Potato Configuration

| Paper Annotation | Potato annotation_type | Notes |
|---|---|---|
| Single-label classification | `radio` | Standard radio buttons |
| Multi-label classification | `multiselect` | Checkboxes |
| Likert/rating scale | `likert` | Set `size`, `min_label`, `max_label` |
| Free-text response | `text` | Open-ended |
| Named entity / span | `span` | With label categories |
| Typed relations between spans | `span_linking` | Requires span scheme first |
| Pairwise preference | `pairwise` | Side-by-side comparison |
| Bounding box | `bbox` or `video_annotation` (tracking) | For images/video |
| Polygon / segmentation | `polygon` | For segmentation masks |
| Video temporal segment | `video_annotation` (segment mode) | Start/end marking |

### Step 3: Create Sample Data

For each task:
1. **If dataset is public**: Download a few examples and adapt them
2. **If dataset is gated**: Create 8-12 synthetic examples that match the schema and domain
3. **Ensure diversity**: Vary the examples across different labels/categories
4. **Include edge cases**: At least 1-2 examples that are ambiguous or represent minority classes

### Step 4: Write Configuration

Follow the config.yaml pattern:
- Set appropriate `annotation_type` for each scheme
- Add `keyboard_shortcuts` for efficiency
- Add `tooltips` with label definitions from the paper
- Use `html_layout` for complex displays (multimodal, pairwise, multi-field)
- Match `text_key` / `context_key` / `audio_key` / `video_key` to sample data fields

## 5. Batch Size and Timing

### Recommended Batch Sizes

- **50 tasks**: Good for a major expansion (like this one). Takes ~15-20 minutes with parallel agents.
- **20 tasks**: Good for targeted gap-filling. Takes ~8-10 minutes.
- **10 tasks**: Good for covering a single new feature or domain.

### Annual Refresh Cadence

Run this process annually after major venue proceedings are published:
- **February**: ACL, EMNLP, NeurIPS, ICLR from previous year are all out
- **August**: NAACL, EACL, CVPR, ICML, INTERSPEECH from current year

## 8. Implementation Notes for Claude Code

### Parallel Agent Strategy

When creating tasks in bulk, organize into groups of 3-10 tasks per agent:
- Group by category (all text/relation-extraction together, all audio together, etc.)
- Launch agents in parallel — 10 agents can run simultaneously
- Each agent creates 3 files per task (metadata.json, config.yaml, sample-data.json)
- Validate all files after all agents complete

### Prompt Template for Task Creation Agents

When instructing agents, provide:
1. Exact directory path for each task
2. The 3-file pattern with field descriptions
3. Paper name, venue, year, annotation types, complexity
4. Brief description of the annotation task
5. Sample data field names
6. Config annotation scheme details (types, labels)
7. Paper URL and dataset URL

### Validation Checklist

After creating tasks:
```bash
# Check all JSON files parse
find . -name "*.json" -path "*/new-task-folder/*" -exec python3 -c "import json,sys; json.load(open(sys.argv[1])); print('OK:', sys.argv[1])" {} \;

# Check all YAML files parse
find . -name "*.yaml" -path "*/new-task-folder/*" -exec python3 -c "import yaml,sys; yaml.safe_load(open(sys.argv[1])); print('OK:', sys.argv[1])" {} \;

# Check sample data has 8+ items
find . -name "sample-data.json" -path "*/new-task-folder/*" -exec python3 -c "import json,sys; d=json.load(open(sys.argv[1])); print(f'{len(d)} items:', sys.argv[1])" {} \;

# Check annotation types in metadata match config
# (manual spot-check recommended)
```

## 9. Quality Standards

### Sample Data Quality
- Each sample-data.json should have 8-12 items
- Items should be realistic and domain-appropriate (not lorem ipsum)
- Include variety across labels/categories
- Include 1-2 edge cases or ambiguous examples
- All items must have an `id` field
- Content fields must match what config.yaml expects

### Metadata Quality
- `paperReference` format: "Author et al., VENUE YEAR"
- `paperUrl` should point to the official proceedings (ACL Anthology preferred for NLP)
- `datasetUrl` should point to the actual dataset (GitHub, HuggingFace, or project page)
- `citation` must be valid BibTeX
- `annotationTypes` must use lowercase Potato type names
- `complexity` must be one of: beginner, intermediate, advanced

### Config Quality
- Every annotation scheme must have `annotation_type`, `name`, `description`, `labels`
- Add `keyboard_shortcuts` for schemes with <15 labels
- Add `tooltips` with brief definitions for each label
- Use `html_layout` for tasks displaying multiple content fields
- Set `allow_all_users: true`
- Point `data_files` to `sample-data.json`
